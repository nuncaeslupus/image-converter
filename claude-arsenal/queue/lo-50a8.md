# Payload: lo-50a8 — T13: Wire traceCache into the trace pipeline

## Acceptance gate
`traceCache.ts` (built + unit-tested in T8/#18) is actually consulted on the
trace path: a retrace whose (imageId, params) was already traced this session
returns the cached SVG without a worker call. Prose gate — verify by a test or
manual check that a repeat of identical params does not re-invoke the worker.

## What the spec requires
Plan "Data flow" step 5: cache keyed by `hash(imageId, params)`, sitting between
the tweak pipeline and the worker call.

## What the code does
`src/lib/traceCache.ts` exists with `traceCacheKey` + `withTraceCache`, fully
unit-tested, but is imported nowhere — the cache is never hit. Root cause: there
is no stable `imageId` in the app. `wizard.image` is a bare `ImageBitmap` with
no identifier, so there is no key to cache on.

## Fix location
Add a stable id to `src/lib/wizard.ts` when an image is set (e.g. a monotonic
counter or content hash), then wrap the worker-trace call in
`src/steps/TraceStep.tsx`'s `runRetrace` with `withTraceCache`. Note the preview
downscale (T11) means the cached input is the downscaled bitmap — key on the
post-downscale identity, or include the cap in the key.
